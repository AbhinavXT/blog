Building safety-critical systems isn't like regular software engineering. When you're working on train protection systems certified to SIL-4 — the highest safety integrity level — the margin for error is essentially zero.

## The Architecture

At its core, our system runs bare-metal on ARM Cortex-R5 processors. No RTOS, no Linux kernel, no safety net of an operating system catching your mistakes. Every clock cycle is accounted for. Every memory access is deliberate.

The communication stack spans multiple protocols: CAN-FD for inter-card communication, UART for debug interfaces, and Ethernet for firmware updates. Each protocol has its own integrity requirements and failure modes.

## Why Bare Metal?

The answer is determinism. When a train is approaching a signal at 130 km/h, you need to guarantee response times in microseconds, not hope for them. An RTOS adds scheduling overhead and non-deterministic behavior that's unacceptable at SIL-4.

## Lessons Learned

The biggest lesson? Defensive programming isn't optional — it's the entire philosophy. Every function validates its inputs. Every state transition is explicitly defined. Every error path is tested. The code might look verbose compared to a typical startup codebase, but verbosity is a feature when lives depend on it.

Writing safety-critical code has fundamentally changed how I think about software. The discipline carries over into everything else I build.
